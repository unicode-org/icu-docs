1. Introduction

As different tools may provide different filters to extract the content of ICU Resource Bundles, it is important for interoperability that they represent the extracted data in identical manner in the XLIFF document.

1.1. Purpose

The intent of this document is to provide a set of guidelines to represent data contained in ICU Resource Bundles as XLIFF content. It offers a collection of recommended mapping of data types in ICU Resource Bundles that developers of XLIFF filters can implement, and users of XLIFF utilities can rely on to ensure a better interoperability between tools.

1.2 Transitional and Strict

XLIFF is specified in two "flavors". Indicate which of these variants you are using by selecting the appropriate schema. The schema may be specified in the XLIFF document itself or in an OASIS catalog. The namespace is the same for both variants. Thus, if you want to validate the document, the tool used knows which variant you are using. Each variant has its own schema that defines which elements and attributes are allowed in certain circumstances.

As newer versions of XLIFF are approved, sometimes changes are made that render some elements, attributes or constructs in older versions obsolete. Obsolete items are deprecated and should not be used even though they are allowed. The XLIFF specification details which items are deprecated and what new constructs to use.

  • Transitional - Applications that produce older versions of XLIFF may still use deprecated items. Use this variant to validate XLIFF documents that you read. Deprecated elements and attributes are allowed.

    xsi:schemaLocation='urn:oasis:names:tc:xliff:document:1.2 xliff-core-1.2-transitional.xsd'

  • Strict - All deprecated elements and attributes are not allowed. Obsolete items from previous versions of XLIFF are deprecated and should not be used when writing new XLIFF documents. Use this to validate XLIFF documents that you create.

    xsi:schemaLocation='urn:oasis:names:tc:xliff:document:1.2 xliff-core-1.2-strict.xsd'

2. Overview of the ICU Resource Bundle file format

An ICU Resource Bundle is a collection of resources. There are two main types of ICU resources: simple and complex. Simple resources each contain a single piece of data. There are five types of simple resources: string, integer, integer vector, binary and alias. Complex resources contain other simple and complex resources. There are two types of complex resources: table and array.

An ICU Resource Bundle consists of a top-level table resource containing all the resources in the bundle. The name of this resource is a locale name.

ICU Resource bundles are specified in a text file. A resource is specified by a name followed by a type name and then the resource data enclosed in { and }.  The following table shows how to specify each resource type:

Resource Type Type Name Resource Data
string :string a text string optionally enclosed in quotes
integer :integer or :int a decimal or hexadecimal integer
integer vector :intvector a comma separated list of integers
binary :binary or :bin a sequence of hexadecimal digits optionally enclosed in quotes
alias :alias a path as a quoted string
table :table one or more named resources
array :array one or more unnamed resources separated by commas

(Notice that all type names start with a colon.)
The following is a detailed explanation of how each resource type is represented in the text file.

2.1 String Resources

String resources are represented by a sequence of Unicode characters, optionally enclosed in quotes. The string must be quoted if it's empty or if it contains a space or any of these special characters: " : , { }.

The string can contain the following escape sequences:
 
Escape Followed By Represents
\u four hex digits a 16 bit Unicode code point
\U eight hex digits a 32 bit Unicode code point
\x two hex digits a Unicode code point between U+0000 and U+00FF
\x{ one to eight hex digits then } a Unicode code point
\ one to three octal digits a Unicode code point between U+0000 and U+00FF
\c any character a control code (low-order four bits of character's code point)
\a ASCII audible alert  (U+0007)
\b ASCII backspace (U+0008)
\e ASCII escape (U+001B)
\f ASCII form feed (U+000C)
\n ASCII newline (U+000A)
\r ASCII carriage return (U+000D)
\t ASCII tab (U+0009)
\v ASCII vertical tab (U+000B)
\ any other character the character (e.g. \\, \", \})

A string resource can be composed of multiple strings separated by whitespace characters. Quoted strings are simply concatenated together. For example, the following two definitions produce the same resource:
processing_image :string {
    "Processing your image. "
    "This may take several minutes."
}


processing_image :string {
    "Processing your image. This may take several minutes."
}
If the strings are not quoted, they are concatenated with a space between them. Again, the following two definitions produce the same resource:
processing_image :string {
    Processing your image.
    This may take several minutes.
}


processing_image :string {
    Processing your image. This may take several minutes.
}
(Note that in this example, each word is actually a separate string because the spaces between them separate the strings.)

String resources may contain variable data represented by a special descriptor enclosed in { and }. If these descriptors are present then either the string must be quoted or the { and } must be escaped.

String resources can also be specified indirectly using the type name :include and a string value that is the name of a file containing the string resource data. The string resource data in this file can contain the same escape sequences as a regular string resource.

2.2 Integer Resources

An integer resource is specified by a sequence of decimal digits or by 0x followed by a sequence of hexadecimal digits. A negative value is proceeded by a minus sign. The value must be between -32,768 and 32,767. (i.e. it must fit in 32 bits.)

2.3 Integer Vector Resources

An integer vector is specified by a sequence of integer values separated by commas. Each integer value is specified the same way that integer resources are specified. The last value may be followed by a comma. The vector may contain a single value, or it may be empty.

2.4 Binary Resources

A binary resource represents a chunk of binary data. It is represented by a sequence of hexadecimal digits, optionally enclosed in quotes. There must be an even number of digits. Quotes must be used to specify an empty resource.

Binary resources can be specified indirectly using the type name :import and a string value that is the name of a file containing the binary data.

2.5 Alias Resources

An alias resource is a reference to another resource, possible in a different resource bundle. The reference is specified as a quoted string.

2.6 Table Resources

A table resource contains one or more named resources, including array resources and other table resources. The contained resources are accessed by name. The top-level resource in an ICU resource bundle is a table resource that contains all of the resources in the bundle. The name of this resource is a locale name.

2.7 Array Resources

An array resource contains a sequence of unnamed resources, separated by commas. The contained resources are a accessed by index rather than by name. If the array contains all string resources, they can be specified without the :string type name or the enclosing { and }. For example:
fish :array {
    "One fish",
    "Two fish",
    "Red fish",
    "Blue fish"
}

2.8 Implied Resource Types

If a type name is not given, the type is determined by looking at what follows the {. A resource name followed by a type name or a { means that the resource is a table. A type name or a { means that the resource is an array. A string followed by a comma means that the resource is an array of strings. A string followed by a } means that the resource is a string.

2.9 Comments

The text file can also contain C++ style comments. That is, comments that start with // and extend to the end of the line, and multi-line comments that start with /* and end with */. There can also by "doc" comments that start with /** and end with */. These comments apply to the following resource and may contain two special tokens. The token @translate followed by yes or no instructs the translator whether or not the resource should be translated. The token @note begins a special not to the translator that extends to the end of the line.

2.10 Sample ICU Resource Bundle

Here is a sample ICU resource bundle containing the English resources for a fictitious Hello World application. For clarity all resource type names have been specified.

/*
 * The resources for a fictitious Hello World application. The application
 * displays a single window with a logo and the hello message.
 */
en :table {
    /**
     * @note This is the message that the application displays to the user.
     */
    hello :string {"Hello, world!"}

    /**
     * The height of the application window.
     *
     * @note Make sure this is tall enough to display the translated message.
     */
    window_height :integer {200}

    /**
     * The width of the application window.
     *
     * @note Make sure this is wide enough to display the translated message.
     */
    window_width :integer {600}

    /**
     * The application version number
     */
    version :intvector {
        /**
         * The major version number.
         *
         * @note The major version number.
         * @translate no
         */
        1,

        /**
         * The minor version number.
         *
         * @translate no
         */
        2,

        /**
         * The patch level.
         *
         * @translate no
         */
        3}

    /**
     * The MD5 checksum of the application.
     *
     * @translate no
     */
    md5_sum :binary {bcfe765be0fdfab22c5f9efd12c52abc}

    /**
     * The logo to be displayed in the application window.
     *
     * @translate no
     */
    logo :import {"logo.gif"}

    /*
     * The Authors. Just use the name from the root bundle.
     */
    authors :alias {"root/authors"}

    /*
     * The translators.
     */
    translators :array {
        :string {"John E. English"},
        :string {"Alan Smithee"},
        :string {"Allen Smythee"}
    }


    /**
     * The application menus.
     *
     * @note Keep the menus and the menu items in this order.
     */
    menus :table {
        file_menu :table {
            name :string {"File"}

            items :array {
                :string {"New"},
                :string {"Open..."},
                :string {"Save"},
                :string {"Save As..."},
                :string {"Exit"}
            }
        }

        edit_menu :table {
            name :string {"Edit"}

            items :array {
                :string {"Cut"},
                :string {"Copy"},
                :string {"Paste"},
                :string {"Delete"}
            }
        }

        help_menu :table {
            name :string {"Help"}

            items :array {
                :string {"Help Topics"},
                :string {"About Hello World"}
            }
        }
    }
}

3. General Considerations

This section discusses the general considerations to take in account when extracting data from ICU resource bundles.  ICU has a tool for converting resource bundle source files to XLIFF and for converting XLIFF files to ICU resource bundle source files. We recommend the use of these tools whenever possible.

3.1. ICU Resource Bundles

The ICU genrb tool converts the resource bundle text files into binary resource files. It is possible to extract data from either the text or the binary files. We recommend extrracting the data from the text files because the comments, and the meta-information that they contain, such as @translate and @note, is not included in the binary files.

Where practical, we recommend maintaining resource data as XLIFF files and converting to ICU format only as a build step. For more information on localization workflow process, please see the Localizing with XLIFF & ICU.

3.2. Source and Target Languages

In ICU resources files the source language is appended to the file name. As an alternative, the filter tools can provide an option for specifying the source language of the file. ICU uses [RFC 4646] for naming its files and resource. The ICU resource format does not provide a way to store both a source and a target language in the resource file. The idea is that developers work on generating the base resource file and convert that file to multiple XLIFF files where the target language is specifed. So, the tools that convert ICU resources to XLIFF must provide an option for specifying the target language of each XLIFF file.

3.3. Translation Unit IDs

ICU RB file format uses ID based translation units just like other major resource formats.

3.4. Character Set Conversion

The source of ICU Resource Bundles can be any one of the major encodings supported by ICU, however it is strongly recommended that the files be stored in UTF-8 encoding.

When extracting data from ICU Resource Bundle files, filters should respect the encoding of the file and provide conversion to UTF-8 for storing data in XLIFF.

4. General Structure

Note: all XLIFF source examples will be shown with extra whitespace to enhance readability. We recommend that filters generate XLIFF files without any extra whitespace to keep the generated files as small as possible.
<?xml version="1.0" encoding="utf-8"?>
<xliff version="1.2" xmlns="urn:oasis:names:tc:xliff:document:1.2">
    <file xml:space = "preserve" source-language = "en" datatype = "x-icu-resourcebundle"
            original = "en.txt" date = "2007-04-11T19:03:09Z">
        <header>
            <tool tool-id = "genrb-icu-3.7.1" tool-name = "genrb" />
        </header>
        <body>
            <group restype = "x-icu-table" id = "en" >
                ... the resources in the resource bundle...
            </group>
        </body>
    </file>
</xliff>

Each ICU Resource Bundle file maps to one XLIFF <file> element. XLIFF representations of ICU Resource Bundle files should have the datatype attribute set to x-icu-resourcebundle, and the original attribute set to the name of the resource bundle file from which the data was extracted.

The optional XLIFF <header> element contains a <tool> element which identifies the tool that extracted the data from the ICU resource bundle.

The XLIFF <body> element contains translation units, which may be grouped using hierarchical <group> elements representing table and array resources..

5. Detailed Mapping

5.1. Comments

Comments in the text file that start with /** and end with */, i.e. "doc" comments, should be extracted and included as comments in the XLIFF file. If the comment contains  @translate, this should be used to set the tranlsate attribute on the first <trans-unit> generated after the comment is extracted. It is not necessary to generate a translate attribute for @translate yes since this is the default.

If the comment contains @note, this should be extracted to a <note> element in the first <trans-unit> generated after the comment is extracted.

For example:

/**
 * These are top level comments for the bundle. Tag name: root
 * @translate yes
 * @note Comments for tag named root
 */
root{
}
would be mapped to: 
    <file xml:space = "preserve" source-language = "en" datatype =
          "x-icu-resourcebundle" original = "root.txt" tool = "genrb" date =
             "2007-01-17T20:58:58Z">
        <body>
            <group restype = "x-icu-table" xml:space = "preserve" id = "ra" >
                <!--These are top level comments for the bundle. Tag name: root-->
                <note> Comments for tag named root</note>
            </group>
        </body>
    </file>

5.2. Strings

ICU provides two ways to specify string resources, namely, string resources with an ID and string resources without an id. The string resources without an ID are used in array resources.

For example:

    /**
     * Tag name: test2
     * @note This resource is for test2
     * @translate yes
     */
    test2:string{"some translatable stuff"} 
    ...
        /**
         * comments for un-named string
         * @translate yes
         * @note This resource is un-named string
         */
        :string{"abcde"}, 
    ...
would be mapped to:
                <trans-unit xml:space = "preserve" id = "test2" resname = "test2" translate="yes">
                    <!--Tag name: test2-->
                    <source >some translatable stuff</source>
                    <note> This resource is for test2</note>
                </trans-unit>
             ... 
                    <trans-unit xml:space = "preserve" id = "<resource name>_<array_incdex>" 
                                translate="yes">
                        <!--comments for un-named string-->
                        <source >abcde</source>
                        <note> This resource is un-named string</note>
                    </trans-unit>
                ...

5.2.1. Preserving Message Replaceables in Value

Messages are a concatenation of strings, numbers, and dates that present a complex formatting challenge—how to put together the sequences of strings, numbers, dates, and other formats to create language-neutral messages. Localization is facilitated because there is no required hard coding message strings or concatenation sequences. ICU has two formats used to create language-neutral messages: MessageFormat and ChoiceFormat. The replaceables in these formats should be enclosed in <ph> elements by the filter that generates the XLIFF representation. These strings are identical to Java's MessageFormat and ChoiceFormat strings, so the handling of these messages is also identical. The filter developers should parse the elements correctly and hide the logic from the translator.

For example:

 choiceFormat:string{"Folder {0} contains {1,choice,0#no files|1#one file|1<{2,number,integer}files}."}
 msgFormat:string{"At {1,time} on {1,date}, there was {2} on planet{0,number,integer}."}

would be mapped to:

<source>Folder <ph id="1">{0}</ph> contains 
  <ph id="2">{1,choice,0#<sub>no files</sub>|1#<sub>one file</sub>|1&lt;{2,number,integer}
             <sub>files</sub>}</ph>.
</source>

<source>At <ph id="1">{1,time}</ph> on <ph id="2">{1, date}</ph>, 
        there was <ph id="3">{2}</ph> on planet <ph id="4">{0,number,integer}</ph>.
</source>

5.3. Integers

ICU provides two ways to specify storage of 32 bit integer resources, namely, integer resources with an ID and integer resources without an id. The integer resources without an ID are used in array resources.

For example:

        /**
         * Tag name: test5
         * @translate yes
         * @note This resource is for test5
         */
        test5:int{1}
    ...
        /**
         * comments for un-named int
         * @translate yes
         * @note This resource is un-named int
         */

        :int{1},
    ...
would be mapped to:
                    <trans-unit restype = "x-icu-int" xml:space = "preserve" id = "test5" resname = "test5"
                                   translate="yes">
                        <!--Tag name: test5-->
                        <source>1</source>
                        <note> This resource is for test5</note>
                    </trans-unit>
               ...
                   <trans-unit xml:space = "preserve" id = "<resource name>_<array_index>" 
                               translate="yes">
                        <!--comments for un-named int-->
                        <source>1</source>
                        <note> This resource is un-named int</note>
                    </trans-unit>
               ...

5.4. Binary Data

Binary data is represented as a string of decimal integers representing the binary values. ICU provides two ways to specify binary resources, binary resources with an ID and binary resources without an id. The binary resources without an ID are used in array resources.

For example:

        /**
         * Tag name: test6
         * @translate no
         * @note This resource is for test6
         */
        test6:bin{"0102"}    ...
        /**
         * comments for un-named binary
         * @translate no
         * @note This resource is for un-named binary
         */
        :bin{"12312312"}
    ...
would be mapped to: 
    <bin-unit restype = "x-icu-bin" id = "test6"  mime-type = "application/octet-stream"
              resname = "test6" translate="no">
        <!--Tag name: test6-->
        <bin-source>
            <internal-file form = "application" crc = "480784534">0102</internal-file>
        </bin-source>
        <note> This resource is for test6</note>
    </bin-unit>
    ...    <bin-unit restype = "x-icu-bin" id = "<resource name>_<array_index>"
              mime-type = "application/octet-stream"
              resname = "<resource name>_<array_index>" translate="no">
        <!--comments for un-named binary-->
        <bin-source>
            <internal-file form = "application" crc = "-2140397846">12312312</internal-file>
        </bin-source>
        <note> This resource is for un-named binary</note>
    </bin-unit>      ...

5.5. Aliases

Aliases point to other resources. They are useful for preventing duplication of data in resources that are not on the same branch of the fallback chain. Alias can also have an empty path. In that case the position of the alias resource is used to find the aliased resource. For a alias resource the translate attribute should always be set to "no".

For example:

        aaa:alias { "/ICUDATA/en/aab" }
would be mapped to: 
    <trans-unit restype = "x-icu-alias" xml:space = "preserve" id = "aaa" resname="aaa" translate="no">
        <source><ph xid="/ICUDATA/en/aab"/></source>
    </trans-unit>

5.6. Arrays

Arrays are used for bundling un-named items together.The ID values for each element of the array are a concatenation of the ID of the enclosing resource and the index of the individual element separated by an underscore. If the enclosing resource is an un-named resource then, the index of the resource should be used.
For example: 
    /**
     * Top level comments for Tag name: TestArray
     * @translate yes
     * @note This resource is for Tag name: Array
     */
    TestArray:array{
        /**
         * comments for un-named string
         * @translate yes
         * @note This resource is un-named string
         */
        :string{"abcde"},
        /** 
         * comments for un-named int
         * @translate yes
         * @note This resource is un-named int
         */
        :int{1},
        /**
         * comments for un-named binary
         * @translate yes
         * @note This resource is for un-named binary
         */
        :bin{"12312312"}
    }

would be converted to: 
             <group restype = "x-icu-array" xml:space = "preserve" id = "TestArray" resname="TestArray">
                 <!--Top level comments for Tag name: TestArray-->
                 <note> This resource is for Tag name: Array</note>
                 <trans-unit xml:space = "preserve" id = "TestArray_0" translate="yes">
                     <!--comments for un-named string-->
                     <source >abcde</source>
                     <note> This resource is un-named string</note>
                 </trans-unit>
                 <trans-unit restype = "x-icu-int" xml:space = "preserve" id = "TestArray_1" translate="yes">
                        <!--comments for un-named int-->
                        <source>1</source>
                        <note> This resource is un-named int</note>
                 </trans-unit>
                 <bin-unit restype = "x-icu-bin" id = "TestArray_2"
                           mime-type = "application/octet-stream" translate="yes">
                     <!--comments for un-named binary-->
                     <bin-source>
                         <internal-file form = "application" crc = "-2140397846">12312312</internal-file>
                     </bin-source>
                     <note> This resource is for un-named binary</note>
                 </bin-unit>
             </group>

5.7. Integer Vectors

Integer vectors are arrays of 32 bit integers and are used for bundling un-named integers together. The ID values for each element of the array are a concatenation of the ID of the enclosing resource and the index of the individual element separated by an underscore. If the enclosing resource is an un-named resource then, the index of the resource should be used.

For example: 
    /**
     * comments for TestIntVector
     * @translate yes
     * @note This resource is for TestIntVector
     */
     TestIntVector{
        /**
         * comments for element 1
         * @translate yes
         * @note This resource is for element 1
         */
         1,
        /**
         * comments for element 2
         * @translate yes
         * @note This resource is for element 2
         */
         2,
        /**
         * comments for element 3
         * @translate yes
         * @note This resource is for element 3
         */
         3
     }

would be converted to: 
  <group restype = "x-icu-intvector" xml:space = "preserve" id = "TestIntVector" resname="TestIntVector">
    <!--comments for TestIntVector-->
    <note> This resource is for TestIntVector</note>
    <trans-unit xml:space = "preserve" id = "TestIntVector_0" translate="yes">
        <!--comments for element 1-->
        <source >1</source>
        <note> This resource is for element 1</note>
    </trans-unit>
    <trans-unit xml:space = "preserve" id = "TestIntVector_1" translate="yes">
        <!--comments for element 2-->
        <source >2</source>
        <note> This resource is for element 2</note>
    </trans-unit>
    <trans-unit xml:space = "preserve" id = "TestIntVector_2" translate="yes">
        <!--comments for element 3-->
        <source >3</source>
        <note> This resource is for element 3</note>
    </trans-unit>
  </group>

5.8.Tables

Tables are arrays of named items and are used for bundling named items together.

For example: 
    /**
     * comments for TestTable
     * @translate yes
     * @note This resource is for TestTable
     */
     TestTable{
        /**
         * comments for Month1
         * @translate yes
         * @note This resource is for Month1
         */
         Month1{"January"}
        /**
         * comments for Month2
         * @translate yes
         * @note This resource is for Month2
         */
         Month2{"February"}
        /**
         * comments for Month3
         * @translate yes
         * @note This resource is for Month3
         */
         Month3{"March"}
     }

would be converted to: 
   <group restype = "x-icu-intvector" xml:space = "preserve" id = "TestTable" resname="TestTable">
        <!--comments for TestTable-->
        <note> This resource is for TestTable</note>
        <trans-unit xml:space = "preserve" id = "TestTable_Month1" translate="yes">
            <!--comments for Month1-->
            <source >January</source>
            <note> This resource is for Month1</note>
        </trans-unit>
        <trans-unit xml:space = "preserve" id = "TestTable_Month2" translate="yes">
            <!--comments for Month2-->
            <source >February</source>
            <note> This resource is for Month2</note>
        </trans-unit>
        <trans-unit xml:space = "preserve" id = "TestTable_Month3" translate="yes">
            <!--comments for Month3-->
            <source >March</source>
            <note> This resource is for Month3</note>
        </trans-unit>
   </group>
·

5.9. Import Directive

Binary data can be imported into a resource with the import data type.

For example:

    /**
     * comments for TestImport
     * @translate no
     * @note This resource is for TestImport
     */
    TestImpport:import{"mytest.jpg"}
would be mapped to: 
   <bin-unit restype = "x-icu-import" id = "TestImpport"
             mime-type = "application/octet-stream" 
             resname = "TestImpport" translate="no">
       <!--comments for TestImport-->
       <bin-source>
              <external-file href = "mytest.jpg"/>
       </bin-source>
       <note> This resource is for TestImport</note>
   </bin-unit>

5.10. Include Directive

Include directive indicates that text data imported into a resource. The contents of the file that is pointed to is replaced in the section.

For example:

    /**
     * comments for TestInclude1
     * @translate no
     * @note This resource is for TestInclude1
     */
    TestInclude1:include{"translit_rules.txt"}

    /**
     * comments for TestInclude2
     * @translate yes
     * @note This resource is for TestInclude2
     */
    TestInclude2:include{"mystring.txt"}
would be mapped to: 
   <bin-unit restype = "x-icu-import" id = "TestInclude1"  mime-type = "application" 
             resname = "TestInclude1" translate="no">
       <!--comments for TestImport1-->
       <bin-source>
              <external-file href = "translit_rules.txt"/>
       </bin-source>
       <note>This resource is for TestInclude1</note>
   </bin-unit>
   <trans-unit xml:space = "preserve" id = "TestInclude2" resname = "TestInclude2" translate="yes">
        <!--comments for TestInclude-->
        <source>Contents of mystring.txt</source>
        <note>This resource is for TestInclude2</note>
   </trans-unit>
